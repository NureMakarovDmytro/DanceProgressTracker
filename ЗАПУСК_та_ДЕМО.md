# Детальна інструкція: запуск коду та демонстрація (ЛР1–ЛР5)

Покрокова інструкція для системи **DanceProgressTracker**: кожна команда, що
з'явиться на екрані після неї, і що саме показувати по кожній лабораторній.

> Усі шляхи нижче — від кореня репозиторію. Спочатку перейдіть у нього:
> ```bash
> cd /Users/dmitromakarov/apz/DanceProgressTracker
> ```

node --version     # очікувано: v18.x або новіше (у нас v25)

## 1. БАЗА ДАНИХ MongoDB (потрібна для backend)

Найпростіше — у Docker (один раз):

```bash
docker run -d --name mongo -p 27017:27017 mongo:7
```

Що з'явиться: довгий ідентифікатор контейнера. Перевірити, що працює:

```bash
docker ps
# У списку має бути рядок зі STATUS "Up ... " та портом 0.0.0.0:27017->27017/tcp
```

Зупинити / знову запустити пізніше:

```bash
docker stop mongo      # зупинити
docker start mongo     # запустити знову
```

## 3. ЛР3 — WEB-ЗАСТОСУНОК

```bash
cd /Users/dmitromakarov/apz/DanceProgressTracker/web
npm install
```

**Варіант А — з реальним backend** (рекомендовано для захисту; backend з п.2 має працювати):

Доступи: **admin / admin** (адміністратор), **teacher / teacher** (викладач).

### Що показати (ЛР3) — покроково

**Частина адміністратора:**
1. Вхід → ввести `admin` / `admin` → «Увійти». З'явиться шапка з вкладками **Групи · Користувачі · Дані**.
2. Вкладка **Групи** → «Додати групу» (назва, стиль, рівень) → зберегти. Показати «Редагувати» та «Видалити».
3. Натиснути «Відкрити» на групі → таблиця учнів з колонками **Пропуски / Середній бал**; «Додати учня».
4. Вкладка **Користувачі** → «Додати користувача» (роль Викладач/Адміністратор).
5. Вкладка **Дані** → «Завантажити копію» (завантажиться `dpt-backup.json`), «Експорт CSV» (`report.csv`), «Обрати файл» для відновлення.
6. **i18n:** угорі праворуч перемкнути **UK → EN**. Показати, що: змінились усі підписи; формат дати став `Jun 8, 2026` замість `8 черв. 2026 р.`; порядок сортування прізвищ відповідає мові.

**Частина викладача:**
7. «Вийти» → увійти як `teacher` / `teacher` (вкладок Користувачі/Дані немає — інший рівень доступу).
8. Відкрити групу → з'являться колонки **Присутній** (чекбокс) і **Оцінка** (1–12).
9. Зняти присутність у когось, виставити оцінки решті → «Зберегти заняття» → з'явиться зелене «Заняття збережено».
10. Показати оновлені **Пропуски** та **Середній бал** у таблиці.

## 4. ЛР2 — МОБІЛЬНИЙ ЗАСТОСУНОК (Android)

> Backend (п.2) має працювати на цьому ж комп'ютері.

1. Відкрити **Android Studio** → *Open* → вибрати теку
   `/Users/dmitromakarov/apz/DanceProgressTracker/mobile`.
2. Дочекатися **Gradle Sync** (статус-бар унизу; перший раз кілька хвилин —
   завантажуються залежності).
3. Створити/запустити **емулятор**: *Device Manager → Create Device* (напр. Pixel,
   API 34) або вибрати наявний.
4. Натиснути **Run ▶** (Shift+F10). Застосунок збереться і відкриється в емуляторі.

Базовий URL уже прописано: `http://10.0.2.2:4000/api/`
(`10.0.2.2` — це localhost комп'ютера з точки зору Android-емулятора; cleartext HTTP дозволено в маніфесті).

> **Реальний телефон** (не емулятор): дізнайтесь IP комп'ютера (`ipconfig getifaddr en0`),
> замініть `10.0.2.2` на цей IP у файлі
> `mobile/app/src/main/java/com/dpt/mobile/network/RetrofitClient.kt`, телефон і ПК — в одній Wi-Fi.

### Що показати (ЛР2) — покроково

1. **Екран входу** → ввести `teacher` / `teacher` → «Увійти».
2. **Список груп** — підвантажується з сервера (ті самі групи, що у вебі).
3. Натиснути на групу → **екран відвідуваності**: перемикачами відмітити
   присутність, у випадному списку обрати оцінки → «Зберегти заняття» → «Заняття збережено».
4. Кнопка «Статистика групи» → **пропуски та середній бал** по учнях.
5. (Сильний хід) переключитися на веб-клієнт і показати, що збережене мобільним
   заняття видно й там — **спільний backend і дані**.

## 5. ЛР4 — МАСШТАБУВАННЯ БЕКЕНДА

Тут backend з п.2 НЕ потрібен — піднімаємо все в Kubernetes (minikube).

### 5.1 Образ бекенда і кластер

```bash
cd /Users/dmitromakarov/apz/DanceProgressTracker

# зібрати Docker-образ бекенда (контекст — тека backend)
docker build -f deploy/docker/Dockerfile -t dpt-backend:latest backend
```
Очікувано: рядки `Successfully built ...` / `naming to docker.io/library/dpt-backend:latest`.

```bash
minikube start                       # підняти локальний кластер
minikube image load dpt-backend:latest   # завантажити образ у кластер
minikube addons enable metrics-server     # потрібно для HPA (автомасштабування)
```

### 5.2 Розгортання

```bash
cd deploy/k8s

kubectl apply -f namespace.yaml
```
Вивід: `namespace/dpt created`

```bash
# Secret з ключем JWT (потрібен бекенду). Mongo-secret НЕ створюємо —
# тоді Mongo підніметься без автентифікації, що збігається з MONGO_URI бекенда.
kubectl -n dpt create secret generic backend-secret \
  --from-literal=jwt-secret='dev-secret-change-me'
```
Вивід: `secret/backend-secret created`

```bash
kubectl apply -f mongo.yaml
kubectl apply -f backend-deployment.yaml -f backend-service.yaml -f backend-hpa.yaml
```
Вивід: `deployment.apps/... created`, `service/backend created`, `horizontalpodautoscaler.apps/backend created`.

```bash
kubectl -n dpt get pods
```
Дочекатися статусу **Running** (1 под mongo + 2 поди backend):
```
NAME                       READY   STATUS    RESTARTS   AGE
backend-xxxxxxxxxx-aaaaa   1/1     Running   0          40s
backend-xxxxxxxxxx-bbbbb   1/1     Running   0          40s
mongo-yyyyyyyyy-ccccc      1/1     Running   0          50s
```

### 5.3 Заповнення бази в кластері (обов'язково перед тестом)

База в кластері порожня. Засідимо її через под бекенда (там є `src/seed.js`):

```bash
kubectl -n dpt exec deploy/backend -- node src/seed.js
```
Вивід: `Seed done` (тепер у кластерній Mongo є `admin/admin`, `teacher/teacher`).

### 5.4 Демонстрація масштабування

У **вікні 1** — спостереження:
```bash
kubectl -n dpt get pods -w        # -w = watch, оновлюється наживо
```
У **вікні 2** — зміна кількості реплік:
```bash
kubectl -n dpt scale deployment/backend --replicas=6
```
У вікні 1 побачите, як з'являються нові поди backend (Pending → ContainerCreating → Running).

Подивитися автоскейлер:
```bash
kubectl -n dpt get hpa
```
Вивід (приклад):
```
NAME      REFERENCE            TARGETS   MINPODS  MAXPODS  REPLICAS
backend   Deployment/backend   12%/60%   2        10       6
```

### 5.5 Навантажувальне тестування (k6)

Прокинути сервіс на localhost (вікно 3):
```bash
kubectl -n dpt port-forward svc/backend 8080:80
```
Вивід: `Forwarding from 127.0.0.1:8080 -> 4000`

Запустити тест (вікно 4):
```bash
cd /Users/dmitromakarov/apz/DanceProgressTracker
k6 run -e BASE_URL=http://localhost:8080 deploy/loadtest/loadtest.js
```
Наприкінці k6 виведе підсумок — ключові рядки:
```
http_reqs..............: 24000   ~200/s        ← це і є RPS
http_req_duration......: p(95)=520ms
http_req_failed........: 0.00%
✓ login 200
```

**Методика:** запустити тест послідовно при `--replicas=2`, `4`, `6`
(`kubectl -n dpt scale deployment/backend --replicas=N` між прогонами) і записати
`http_reqs (RPS)` та `p(95)` для кожного — RPS зростає з кількістю подів.

### 5.6 Прибирання

```bash
kubectl delete namespace dpt    # видалити все
minikube stop                   # зупинити кластер
```

### Що показати (ЛР4)

1. `kubectl get pods` — стартові **2 поди** backend.
2. `kubectl scale --replicas=6` → у `get pods -w` поди ростуть до **6**.
3. `kubectl get hpa` — автоматичне масштабування за CPU (ціль 60%).
4. `k6 run` для 2 і для 6 подів → показати, що **RPS (http_reqs/s) зростає**.
5. Пояснити вузьке місце (графік у звіті): RPS росте майже лінійно, поки
   обмежувач — CPU подів, і виходить на плато, коли ним стає **єдина MongoDB**.

## 8. Типові проблеми

| Симптом | Причина / рішення |
|---------|-------------------|
| backend: `Startup failed: ... ECONNREFUSED 27017` | Не запущено Mongo. Виконайте `docker start mongo` (п.1). |
| web: вхід не проходить, помилка мережі | У режимі реального API не запущено backend, або `web/.env` без `VITE_API_URL`. Запустіть backend або приберіть `.env` (демо-режим). |
| мобільний: «Сталася помилка» / немає груп | Backend не працює, або на реальному телефоні не замінено `10.0.2.2` на IP комп'ютера. |
| мобільний: не збирається | У Android Studio: *File → Sync Project with Gradle Files*; перевірте наявність JDK 17 (Settings → Build Tools → Gradle). |
| ЛР4: поди backend у `CrashLoopBackOff` | Не створено `backend-secret`. Виконайте команду створення секрета (п.5.2). |
| ЛР4: k6 «Не вдалося залогінитись» | Не засіяно кластерну базу. Виконайте `kubectl -n dpt exec deploy/backend -- node src/seed.js` (п.5.3). |
| ЛР4: HPA показує `<unknown>/60%` | Не увімкнено metrics-server: `minikube addons enable metrics-server`, зачекати ~1 хв. |
| ЛР4: `ImagePullBackOff` | Образ не завантажено в кластер: `minikube image load dpt-backend:latest`. |
