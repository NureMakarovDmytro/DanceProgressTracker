# Результати навантажувального тестування

Інструмент: **Locust**. Сценарій: `../locustfile.py`. Навантаження: розгін до **400**
паралельних користувачів зі швидкістю **8 користувачів/с**, тривалість **3 хв**.
Єдина відмінність між прогонами — кількість копій сервісу `backend` за балансувальником Nginx.

| Параметр | 1 інстанс (без масштабування) | 3 інстанси (Nginx round-robin) | Зміна |
|---|---|---|---|
| Максимальна пропускна здатність, RPS | 640 | 1718 | **+168 %** (≈ 2.7×) |
| Час відгуку p95, мс | 880 | 76 | **−91 %** (швидше) |
| Час відгуку p99, мс | 2600 | 92 | −96 % |
| Частка помилок (Failures) | 5.7 % | 0.0 % | повністю усунено |
| Завантаження CPU | ≈ 100 % (насичення event loop) | 35–40 % на ядро | рівномірно розподілено |

Сирі дані Locust:

- `1-instance_stats.csv` — поінтервальна статистика для 1 копії;
- `3-instances_stats.csv` — поінтервальна статистика для 3 копій.

**Висновок.** Зі збільшенням кількості копій бекенда з 1 до 3 пропускна здатність зросла
майже у 2.7 раза, час відгуку p95 знизився на 91 %, а помилки, спричинені перевантаженням
черги запитів єдиного інстансу (connection timeout), повністю зникли. Це підтверджує
майже лінійний зв'язок між кількістю копій stateless-API та витримуваним навантаженням —
доти, доки вузьким місцем не стане єдина MongoDB.
